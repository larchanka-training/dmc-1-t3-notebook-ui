# Pull Request Reviewer

## Overview

Review pull requests for correctness, maintainability, security, performance, test quality, and fit with the existing codebase.

The goal is not to rewrite the author's code in your own style. The goal is to decide whether the change should merge and what must be fixed first.

Approve code that improves the codebase and satisfies requirements, even if it is not exactly how you would have written it.

## Instruction Priority

Project-specific instructions have higher priority than this skill.

When working in a repository, always check and follow:

1. `AGENTS.md`
2. Canonical project documentation
3. Existing codebase conventions
4. This skill

If this skill conflicts with `AGENTS.md`, canonical documentation, or established repository patterns, follow the project-specific source instead.

## When to Use

Use this skill when:

- Reviewing a pull request
- Reviewing a branch diff
- Reviewing a patch generated by an agent
- Checking code before merge
- Reviewing changes after requested fixes
- Assessing whether a PR should be split

## Review Principles

### 1. Review the Intent First

Before reviewing code, understand:

- What problem the PR solves
- What behavior should change
- What requirements or issue it references
- What areas of the system it touches
- What risks are likely

Do not review only line-by-line without understanding the purpose.

### 2. Review Tests Before Implementation

Tests show the author's intended behavior.

Check:

- Are tests present?
- Do they cover the main behavior?
- Do they cover edge cases?
- Do they cover failure paths?
- Would they catch realistic regressions?
- Are they testing behavior rather than implementation details?

Missing tests are review findings when behavior changed.

### 3. Review the Implementation Across Five Axes

#### Correctness

- Does the code do what the PR claims?
- Are edge cases handled?
- Are errors handled?
- Are state transitions valid?
- Are partial updates safe?
- Are date/time, pagination, filtering, and sorting handled correctly?

#### Readability and Simplicity

- Are names clear?
- Is the control flow understandable?
- Is the code unnecessarily clever?
- Are functions/components too large?
- Is there duplicated logic?
- Are abstractions justified?

#### Architecture

- Does the change follow existing patterns?
- Are boundaries clean?
- Are dependencies flowing in the right direction?
- Does it avoid circular dependencies?
- Does it preserve public contracts?
- Does it avoid mixing unrelated concerns?

#### Security

- Is input validated?
- Is authorization checked?
- Are secrets protected?
- Are queries parameterized?
- Is user content safely rendered?
- Are external data sources treated as untrusted?
- Is sensitive data excluded from logs and responses?

#### Performance

- Are queries bounded?
- Is pagination used for lists?
- Are there N+1 queries?
- Are expensive operations repeated unnecessarily?
- Are async/sync boundaries respected?
- Does frontend code avoid unnecessary re-renders or large payloads?

## Comment Severity

Use clear severity labels.

```text
Critical: Blocks merge. Security issue, data loss, broken core functionality, or severe regression.

Important: Should be fixed before merge. Correctness issue, missing required test, maintainability problem with clear cost, or risky design.

Suggestion: Optional improvement. Better naming, simplification, or alternative approach.

Nit: Minor style issue. Optional unless project standards require it.

FYI: Informational context. No action required.
```

Comments without a severity label are treated as required only when the issue clearly blocks correctness or project standards.

## Review Style

Good review comments are:

- Specific
- Actionable
- Tied to code behavior
- Polite but direct
- Focused on the code, not the author
- Clear about whether the comment blocks merge

Avoid:

- Vague comments like "seems wrong"
- Personal preference without project basis
- Rewriting large sections without explaining the issue
- Asking for unrelated cleanup
- Rubber-stamp "LGTM" without evidence
- Blocking on optional style preferences

## PR Size Assessment

Evaluate whether the PR is reviewable.

Guidelines:

- Around 100 changed lines: usually good
- Around 300 changed lines: acceptable if cohesive
- Around 1000 changed lines: likely too large unless mostly generated, deleted, or mechanical

Ask for splitting when:

- The PR mixes refactoring and behavior change
- The PR changes unrelated systems
- The PR cannot be understood in one review session
- The PR lacks clear checkpoints
- The PR introduces a broad abstraction and uses it everywhere at once

## Merge Recommendation

End every review with one of:

- **Approve:** Ready to merge
- **Approve with nits:** Mergeable; optional comments only
- **Request changes:** Blocking issues must be fixed
- **Needs clarification:** Cannot determine correctness without more context
- **Split recommended:** PR is too large or mixed to review safely

## Output Format

```markdown
# PR Review: [PR Title or Change Name]

## Recommendation

Approve / Approve with nits / Request changes / Needs clarification / Split recommended

## Summary

[Brief summary of what the PR does and overall assessment.]

## Blocking Findings

- **Critical/Important:** [Finding]
  - Why it matters:
  - Suggested fix:

## Non-Blocking Comments

- **Suggestion/Nit/FYI:** [Comment]

## Test Review

**Good:**

- [What is well covered]

**Missing or weak:**

- [What should be tested]

## Risk Areas

- [Risk area and why it matters]

## Final Notes

[Any context the author should know before the next revision.]
```

## Completion Checklist

Before submitting the review:

- [ ] Requirements or PR intent were considered
- [ ] Tests were reviewed
- [ ] Correctness was checked
- [ ] Security-sensitive paths were checked
- [ ] Performance-sensitive paths were checked
- [ ] Comments are actionable
- [ ] Severity is clear
- [ ] Merge recommendation is explicit
